# 設計力テスト — 設計書

## コンセプト

ランダムに出題される「お題 × 現場条件 × 納期」に対して、エンジニアが設計判断を回答。
AIが5問の対話で深掘りし、設計力スコアを算出する。

正解がない問題に対して「状況に応じた判断ができるか」を測るテスト。

## ユーザーフロー

```
トップページ → 「テスト開始」→ お題＆条件提示 → 5問の対話 → 評価結果
```

1. ユーザーが「テスト開始」を押す
2. お題・現場条件・納期がランダムに組み合わされて提示される
3. AIが5問の質問で設計判断を深掘り
4. 全回答を評価し、設計力スコア＆フィードバックを表示

## お題（100問プール）

JSON で管理。カテゴリごとに用意する。

### カテゴリ例

| カテゴリ | お題例 |
|---------|-------|
| Web アプリ | AIチャットツール、ECサイト、SNS、管理画面 |
| API / バックエンド | 決済API、リアルタイム通知、ファイルアップロード基盤 |
| インフラ / DevOps | CI/CDパイプライン、マイクロサービス移行、監視基盤 |
| データ | レコメンドエンジン、ログ分析基盤、検索システム |
| モバイル | オフライン対応アプリ、プッシュ通知基盤、位置情報アプリ |

### データ構造

```typescript
type Topic = {
  id: string;
  category: "web" | "api" | "infra" | "data" | "mobile";
  title: string;        // 例: "AIチャットツールを設計しろ"
  description: string;  // お題の詳細（2〜3行）
};
```

## ランダム条件

お題とは別に、現場条件・納期・予算をランダムで付与する。

### 条件データ

```typescript
type Constraint = {
  teamSize: string;     // 例: "エンジニア2人"
  company: string;      // 例: "スタートアップ（シード期）"
  deadline: string;     // 例: "2週間"
  budget: string;       // 例: "月5万円以内"
  extra?: string;       // 例: "既存のRailsモノリスあり"
};
```

### 条件パターン例

| 現場 | 納期 | 予算 | 追加制約 |
|------|------|------|---------|
| スタートアップ / 2人 | 2週間 | 月5万以内 | なし |
| 大企業 / 10人チーム | 半年 | 潤沢 | セキュリティ審査あり |
| 受託 / 3人 | 1ヶ月 | 固定予算 | レガシーシステム連携必須 |
| 個人開発 / 1人 | 期限なし | 無料枠のみ | なし |
| メガベンチャー / 5人 | 3ヶ月 | 中規模 | マイクロサービス前提 |

## 質問フェーズ（5問）

お題＆条件を踏まえて、AIが順番に質問する。

| # | カテゴリ | 聞くこと |
|---|---------|---------|
| Q1 | 技術スタック選定 | 何を使う？なぜその技術を選んだ？ |
| Q2 | アーキテクチャ設計 | 全体構成は？コンポーネント分割は？ |
| Q3 | データ設計 | DB選定、スキーマ、状態管理はどうする？ |
| Q4 | 非機能要件 | スケーラビリティ・セキュリティ・パフォーマンスの考慮は？ |
| Q5 | 運用・コスト | デプロイ戦略、監視、予算内に収まるか？ |

### 質問のポイント

- お題と条件に応じて質問内容を動的に変える
- 「スタートアップ×2週間」なら割り切り判断を重視
- 「大企業×半年」ならガチ設計の深さを重視
- **Why を徹底的に突く** — 「何を使うか」より「なぜそれを選んだか」を重視
- 各質問で深掘りを **最大2回** まで許可（戦闘力診断より多い）
- 回答が浅い場合はやさしく、だが容赦なく「なぜ？」を繰り返す

### 深掘りの方針

VPoPとして「意思決定の根拠」を見る。技術知識そのものより判断力を評価。

- 「それ、他の選択肢と比べてなぜそっちにしました？」
- 「その判断、チームに説明するときどう伝えます？」
- 「仮にその技術が使えなかったら、次の候補は？」
- 「それ、3ヶ月後にスケールしたときも同じ判断します？」

曖昧な回答（「なんとなく」「流行ってるから」）には特に深く切り込む。
逆に、根拠が明確な回答にはサクッと次に進む。

### 設計の一貫性チェック

設計は仮説。Q1〜Q5の回答に **矛盾や破綻** がないかをAIが常にチェックする。

- Q1で「スタートアップだから軽量にいく」→ Q3で「RDBを3つ使う」→ 矛盾
- Q1で「Vercel + サーバーレス」→ Q4で「WebSocket常時接続」→ 破綻
- Q2で「モノリス」→ Q5で「マイクロサービスごとにデプロイ」→ 矛盾

破綻を検知したら即座に指摘する:
- 「あれ、さっき軽量にいくって言ってましたよね？DB3つは軽量ですか？」
- 「ちょっと待ってください。Q1の設計方針とズレてきてません？」

**一貫した設計思想を持てるか** が評価の大きなウェイトを占める。

### 回答タイマー（内部制御・ユーザー非表示）

ユーザーには見せないが、回答に一定時間（例: 3分）経過したら AI 側から促す。

- 実務でも設計判断に無限に時間はかけられない
- タイマー超過時: 「考え中ですか？途中まででもいいので、今の時点の考えを聞かせてください」
- 長考自体は減点しないが、**判断スピードも評価の参考値** として記録する
- フロントエンドで各質問の回答時間を計測し、Server Action に送信する

```typescript
type AnswerMeta = {
  questionIndex: number;
  responseTimeMs: number;  // 回答にかかった時間
};
```

## 評価（RAG で強化）

### スコア構成

```typescript
type EvaluationResult = {
  totalScore: number;          // 0〜100
  rank: string;                // ランク名
  scores: {
    stackSelection: number;    // 技術選定力（0〜100）
    architecture: number;      // 設計力（0〜100）
    dataDesign: number;        // データ設計力（0〜100）
    nonFunctional: number;     // 非機能要件の考慮（0〜100）
    operationCost: number;     // 運用・コスト意識（0〜100）
  };
  comment: string;             // 総合コメント
  strengths: string[];         // 良かった点
  improvements: string[];      // 改善点
  modelAnswer: string;         // この条件での模範設計（概要）
};
```

### ランク表

| スコア | ランク |
|--------|--------|
| 0〜29 | ジュニア設計者 |
| 30〜49 | ミドル設計者 |
| 50〜69 | シニア設計者 |
| 70〜84 | リードアーキテクト |
| 85〜100 | チーフアーキテクト |

### RAG ナレッジベース（evaluation のみ）

評価時にユーザーの回答内容でベクトル検索し、適切な評価基準を取得する。

```
src/data/knowledge/evaluation/
├── web-app.md          # Webアプリ設計の評価基準
├── api-backend.md      # API設計の評価基準
├── infrastructure.md   # インフラ設計の評価基準
├── data-system.md      # データ基盤の評価基準
├── mobile.md           # モバイル設計の評価基準
├── startup.md          # スタートアップ条件での評価基準
├── enterprise.md       # 大企業条件での評価基準
└── cost-optimization.md # コスト最適化の評価基準
```

お題と条件の組み合わせごとに「何が良い設計判断か」の基準が変わるため、
RAGで条件に合った評価基準を引っ張ることで、AIの評価のブレを抑える。

## デザインコンセプト

**「インテリ × ポップ」— 知的だけど堅すぎない、エンジニア試験の新しい形**

### トーン & ムード

- 技術系資格試験のような「ちゃんとしてる感」がベース
- でも堅苦しい検定サイトにはしない。遊び心とポップさを混ぜる
- 「頭いい人が作った、でも肩肘張らずに受けられる試験」が理想

### カラーパレット

| 用途 | 色 | 備考 |
|------|-----|------|
| 背景 | `#FAFAF9`（ウォームグレー） | 白すぎず目に優しい、紙のような質感 |
| プライマリ | `#6366F1`（インディゴ） | 知的さ。ボタン・アクセントに使用 |
| セカンダリ | `#F59E0B`（アンバー） | ポップさ。スコア・バッジ・ハイライトに使用 |
| テキスト | `#1C1917`（ストーングレー） | 読みやすく落ち着いた黒 |
| サブテキスト | `#78716C` | 補足情報・ラベル |
| サーフェス | `#FFFFFF` | カード・入力欄の背景 |
| ボーダー | `#E7E5E4` | 薄く上品な区切り線 |
| 成功 | `#10B981`（エメラルド） | 高評価・合格ラインの表現 |
| 警告 | `#EF4444`（レッド） | 低評価・注意の表現 |

### タイポグラフィ

- 見出し: **太め・やや大きめ**。試験タイトルの存在感を出す
- 本文: 読みやすいサイズ感。長文の設計回答を書くので可読性重視
- コード・技術用語: `monospace` で視覚的に区別
- フォント: システムフォント（Noto Sans JP 系）。日本語の可読性を最優先

### UIパターン

- **カード中心のレイアウト**: お題・チャット・結果すべてカードで区切る
- **プログレスバー**: Q1〜Q5 の進捗を上部にポップに表示（ステップインジケーター）
- **カテゴリバッジ**: お題カテゴリを色付きバッジで表示（Web=青、API=緑、インフラ=紫 等）
- **スコア表示**: 数字ドーン + ランク名。試験結果の「出た！」感を演出
- **レーダーチャート**: 5軸スコアをビジュアルに。設計力の偏りが一目でわかる
- **アニメーション**: メッセージのフェードイン、スコアのカウントアップ等。やりすぎず品よく

### やらないこと

- ダークモード（v1では対応しない。ライトモードの完成度を優先）
- 過度な装飾・グラデーション（インテリ感が損なわれる）
- キャラクターのイラスト・アバター（テキストの力で勝負）
- ゲーミフィケーション過多（試験としての信頼感を保つ）

## キャラクター

**イケてるベンチャーのVPoP（VP of Product）**

やさしげな物腰の裏に冷たさがある。褒めてるようで詰めてくる。
笑顔で「なるほどね〜」と言いながら、次の一言で急所を突く。

### 口調・性格

- 基本は丁寧でフレンドリー。だが質問の切れ味が鋭い
- 「いいですね〜」の後に「ただ、」が来る恐怖
- 曖昧な回答にはやさしく微笑みながら容赦なく深掘り
- 本当に良い回答には素直に感心する（ギャップ）

### セリフ例

- 導入: 「はじめまして！今日はカジュアルに設計の話しましょ〜。気楽にいきましょう」
- 深掘り: 「あ〜なるほどね。ちなみにそれ、ユーザー1万人になっても同じ判断します？」
- やんわり否定: 「いい視点ですね。ただ、この条件だとちょっと厳しくないですか？」
- 冷たい一言: 「うん、まあ…動くとは思いますよ。動くだけなら」
- 高評価: 「お、いいですね。ちゃんと現場見てきた人の判断だ」
- 低評価: 「なるほど〜…ちょっと一回整理しません？」
- 結果発表: 「お疲れさまでした！率直にフィードバックしますね」

## API コスト見積もり

| 項目 | 回数 | 概算 |
|------|------|------|
| 対話（5問 × 往復） | 最大10回/セッション | claude-sonnet: 約$0.01/セッション |
| 評価（RAG + 最終評価） | 1回/セッション | claude-sonnet: 約$0.005/セッション |
| Vector 検索 | 1〜2回/セッション | Upstash 無料枠内 |

1日100セッション想定: 約$1.5/日 → 月$45程度
