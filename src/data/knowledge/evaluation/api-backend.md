# API・バックエンド設計の評価基準

## 概要

API設計およびバックエンドアーキテクチャにおける技術的判断力を評価するための基準。
クライアントとの契約としてのAPI設計、ビジネスロジックの構造化、外部連携の設計力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- REST / GraphQL / gRPC をユースケースに応じて使い分けている
- フレームワーク選定時に、エコシステムの成熟度・チームの習熟度を考慮している
- メッセージキュー（SQS, RabbitMQ等）の導入判断が非同期処理の要件に基づいている
- 認証基盤（OAuth2.0, OIDC）の選定がセキュリティ要件を満たしている

### アンチパターン

- すべてのAPIをRESTで設計し、GraphQLやgRPCの検討をしない
- マイクロサービスが不要な規模でマイクロサービスを採用する
- 言語・フレームワーク選定がチームのスキルセットと乖離している
- 外部APIとの連携でSDKの有無やレート制限を確認していない

### スコアリング指針

- **5点**: 通信パターン・チーム構成・将来の拡張性を総合判断し、最適な技術を選定
- **4点**: 要件に合った技術を選定し、主要なトレードオフを説明できる
- **3点**: 一般的に妥当な選定だが、代替案との比較が不足
- **2点**: 選定根拠が「好み」や「経験」のみに依存
- **1点**: 要件と技術の不一致が明らか

---

## 2. 設計力（architecture）

### 良い設計判断

- APIバージョニング戦略（URL, Header, Content Negotiation）を明示している
- エンドポイントの粒度がリソース指向で、一貫した命名規則に従っている
- 冪等性の設計（特にPOST/PUT）を考慮し、リトライ安全性を確保している
- エラーレスポンスのフォーマットが統一され、クライアントが適切にハンドリングできる
- ドメイン駆動設計の概念（集約, 境界づけられたコンテキスト）を適用している

### アンチパターン

- CRUDの羅列でしかなく、ビジネスロジックの所在が不明
- 1つのエンドポイントに複数の責務を持たせる（God Endpoint）
- エラーコードが統一されておらず、クライアント側のハンドリングが困難
- トランザクション境界を考慮せず、データ整合性が担保できない
- バリデーションがコントローラ層に散在し、ドメイン層で保証されない

### スコアリング指針

- **5点**: リソース設計・エラー設計・バージョニングが一貫し、拡張性の高いAPI設計
- **4点**: RESTful原則に従い、主要な設計パターンを適用
- **3点**: 基本的なCRUD設計は適切だが、複雑なユースケースへの対応が不十分
- **2点**: エンドポイント設計に一貫性がなく、クライアントが使いにくい
- **1点**: API設計の基本原則が理解できていない

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- リクエスト/レスポンスのスキーマをOpenAPI仕様で定義している
- ページネーション方式（カーソル vs オフセット）をデータ特性に応じて選択
- バルク操作のAPI設計でパフォーマンスと整合性のバランスを取っている
- イベント駆動設計でのメッセージスキーマにバージョニングを適用している
- 外部キーの参照整合性とアプリケーション層での整合性チェックを使い分けている

### アンチパターン

- レスポンスにデータベースのカラム名をそのまま露出する
- N+1問題が発生するAPIの設計（関連リソースの取得が非効率）
- ページネーションなしで大量データを返すエンドポイント
- 入力値のバリデーションが甘く、不正データが永続化される
- 機密情報（パスワードハッシュ等）がレスポンスに含まれる

### スコアリング指針

- **5点**: スキーマ設計が堅牢で、データの整合性・安全性・効率性を包括的に考慮
- **4点**: 入出力の型定義が明確で、主要なデータパターンに対応
- **3点**: 基本的なスキーマ設計はできるが、エッジケースの考慮が不足
- **2点**: データの型定義が曖昧で、不正なデータが混入する可能性がある
- **1点**: スキーマ設計の概念がなく、フリーフォームでデータをやり取り

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- レート制限の設計（トークンバケット、スライディングウィンドウ等）を要件に応じて実装
- タイムアウト・リトライ・サーキットブレーカーのパターンを適用している
- 認証・認可の多層防御（API Gateway + アプリケーション層）を設計
- CORS設定を適切に行い、不要なオリジンからのアクセスをブロック
- ヘルスチェックエンドポイントの設計で、依存サービスの状態も確認

### アンチパターン

- レート制限がなく、DDoS攻撃や誤った大量リクエストに対して脆弱
- 外部API呼び出しにタイムアウトを設定していない
- 認証トークンの有効期限やリフレッシュ戦略が未設計
- SQLインジェクション対策がORMに依存しきっている
- ログにセンシティブ情報（トークン、パスワード等）が記録される

### スコアリング指針

- **5点**: セキュリティ・耐障害性・パフォーマンスを多層的に設計し、具体的な数値目標を設定
- **4点**: 主要な非機能要件を把握し、適切な対策を設計に組み込んでいる
- **3点**: セキュリティの基本は押さえているが、耐障害性の設計が不足
- **2点**: 非機能要件への言及が断片的で、体系的な対策がない
- **1点**: 非機能要件が全く考慮されていない

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- 構造化ログの設計で、リクエストIDによるトレーシングが可能
- APIドキュメントの自動生成（OpenAPI / Swagger）をCI/CDに組み込んでいる
- Blue/Greenデプロイやカナリアリリースでの段階的デプロイ戦略
- バックエンドのスケーリング戦略（水平スケール / 垂直スケール）を明確にしている
- 障害時のフォールバック戦略（縮退運転、キャッシュ応答）を事前に設計

### アンチパターン

- ログの粒度が粗すぎて、障害時の原因特定に時間がかかる
- デプロイ時のダウンタイムを許容し、ゼロダウンタイムデプロイを考慮しない
- オートスケールの設定がなく、トラフィック増加に手動対応
- API変更時の後方互換性を維持せず、クライアントが突然壊れる
- コスト見積もりがなく、本番運用後に予想外の請求が発生する

### スコアリング指針

- **5点**: 運用の自動化・可観測性・コスト管理を統合的に設計し、チーム運用を見据えている
- **4点**: CI/CD・ログ・監視を含む運用設計があり、コスト感覚もある
- **3点**: 基本的な運用は考慮しているが、障害対応やコスト管理が不十分
- **2点**: 運用設計がほぼなく、手動作業が多い
- **1点**: 運用やコストへの配慮が全くない

---

## 良い回答例

> 「REST APIをOpenAPI 3.1で設計し、スキーマファーストで開発を進めます。
> エンドポイントはリソース指向で `/api/v1/users/{id}/orders` のように階層化します。
> 冪等性キーをヘッダーで受け取り、重複リクエストを安全に処理します。
> レート制限はトークンバケットで実装し、429レスポンスにRetry-Afterヘッダーを付与します。
> 構造化ログにリクエストIDを含め、分散トレーシングで問題箇所を特定できるようにします。」

## 悪い回答例

> 「ExpressでAPIを作って、MySQLからデータを返します。認証はJWTでやります。」
> → 設計の深さがなく、非機能要件や運用面の考慮が欠如
