# コスト最適化の評価基準

## 概要

システム設計におけるコスト最適化の判断力を評価するための基準。
初期コスト・運用コスト・人的コストを総合的に考慮し、ビジネス価値に対して
適切なコスト配分ができる設計力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- OSS vs 商用ライセンスの判断をTCOベースで行っている
- マネージドサービスの採用判断を、自前運用の人件費と比較して行っている
- サーバーレスの採用をトラフィックパターン（スパイク型 vs 定常型）から判断
- 無料枠・クレジットプログラムを把握し、初期フェーズで最大限活用
- マルチクラウドのコスト比較を実施し、ワークロードに最適なプロバイダーを選定

### アンチパターン

- 「無料だから」という理由だけでOSSを選定し、学習・運用コストを無視する
- マネージドサービスの料金体系を理解せず、想定外の課金が発生する
- トラフィックが少ないのにReserved Instancesを購入する
- SaaS/PaaSの契約を見直さず、使っていないサービスに課金し続ける
- コスト比較をせず「前回と同じ構成」で設計する

### スコアリング指針

- **5点**: TCOを算出し、ビジネスフェーズごとの最適なコスト戦略を提示
- **4点**: 主要なコスト要因を把握し、適切な技術選定でコストを抑制
- **3点**: コストは意識しているが、具体的な数値比較が不足
- **2点**: コスト感覚が乏しく、高コストな構成を安易に選定
- **1点**: コストを全く考慮していない

---

## 2. 設計力（architecture）

### 良い設計判断

- コストを考慮したアーキテクチャ（サーバーレス、エッジコンピューティング）を適材適所で採用
- キャッシュ戦略で重複処理・通信を削減し、計算リソースとAPI呼び出しコストを最適化
- 非同期処理でピーク負荷を平準化し、プロビジョニングコストを削減
- マイクロサービスの粒度をコスト効率の観点でも判断している
- 段階的アーキテクチャ（モノリス→モジュラーモノリス→サービス分割）でコストを段階的に投入

### アンチパターン

- 全リクエストをサーバーレスで処理し、高トラフィック時にコストが爆発
- キャッシュを使わず、同一データを何度もDBから取得する
- 不要なリアルタイム処理を設計し、常時稼働のリソースが発生
- マイクロサービスを過度に分割し、サービス間通信のコストが増大
- CDNを活用せず、オリジンサーバーに全トラフィックが集中

### スコアリング指針

- **5点**: コスト効率を意識したアーキテクチャ設計で、トラフィックパターンに最適化
- **4点**: キャッシュ・非同期処理・CDNを活用し、主要なコスト削減を実現
- **3点**: 基本的な設計は妥当だが、コスト最適化の視点が一部不足
- **2点**: コスト非効率な設計が複数あり、無駄な支出が見込まれる
- **1点**: コストを全く考慮しないアーキテクチャ設計

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- ストレージ階層化（ホット / ウォーム / コールド / アーカイブ）でコストを最適化
- データ圧縮形式（Parquet, ORC等）の選定でストレージとクエリコストを削減
- TTL（Time To Live）の設定でデータの自動削除によりストレージコストを管理
- クエリの最適化（パーティショニング、インデックス設計）でスキャン量を削減
- データベースのスケーリング方式（Read Replica, Sharding）をコスト効率で選択

### アンチパターン

- 全データを最高性能のストレージに保存し、コストが膨張する
- 不要なデータを削除せず、ストレージ使用量が増加し続ける
- フルテーブルスキャンが頻発し、クエリコスト（BigQuery等）が高額になる
- バックアップデータを全期間保持し、ストレージコストが際限なく増加
- ログデータの保持期間を設定せず、大量の不要ログにコストを支払い続ける

### スコアリング指針

- **5点**: ストレージ階層化・圧縮・TTL・クエリ最適化を統合したコスト効率の高いデータ設計
- **4点**: 主要なデータコスト要因を把握し、最適化策を実施
- **3点**: 基本的なストレージ管理はあるが、クエリコストや階層化の考慮が不足
- **2点**: データコストの意識が低く、無駄な保存・処理がある
- **1点**: データコストを全く考慮していない

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- 可用性とコストのトレードオフを明示し、事業要件に見合ったSLAを設定
- マルチAZ構成のコスト増を把握し、必要なコンポーネントのみに適用
- オートスケーリングの上限を設定し、予期せぬコスト増を防止
- セキュリティ対策のコストを可視化し、リスクベースで優先順位を決定
- パフォーマンス目標を「コストに見合う」レベルで設定し、過剰投資を回避

### アンチパターン

- 99.999%の可用性を目標にして、コストが天文学的に膨らむ
- オートスケーリングに上限を設けず、DDoS攻撃時にコストが爆発
- 全環境（dev / staging / prod）を同一スペックで構成する
- 夜間や休日に不要なリソースを停止する仕組みがない
- 性能改善に際限なくリソースを投入し、費用対効果が低下

### スコアリング指針

- **5点**: 可用性・性能・セキュリティのコストトレードオフを明示し、最適なバランスを設計
- **4点**: 主要なコストトレードオフを把握し、現実的な目標を設定
- **3点**: 一部のトレードオフは考慮しているが、網羅性が不足
- **2点**: コストを考慮せずに高い目標を設定し、現実的でない
- **1点**: コストとのトレードオフの概念が欠如

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- コスト監視ダッシュボードを設計し、異常検知で即座にアラート
- 月次コストレビューのプロセスを定義し、継続的な最適化を実施
- リソースのタグ付け戦略で、プロジェクト・部門ごとのコスト按分を実現
- 開発環境の自動停止（夜間・休日）でコストを30-50%削減
- スポットインスタンス / プリエンプティブVMの活用でバッチ処理コストを削減

### アンチパターン

- コスト確認が月末の請求書だけで、リアルタイムの把握ができない
- タグ付けルールがなく、どのプロジェクトがコストを消費しているか不明
- 開発・検証環境が24時間365日稼働し、無駄なコストが発生
- Reserved Instancesの未使用分が放置されている
- コスト削減を「いつかやる」タスクとして後回しにし続ける

### スコアリング指針

- **5点**: 継続的コスト最適化のプロセス・自動化・可視化を統合した運用設計
- **4点**: コスト監視とタグ付けが整備され、定期的な見直しプロセスがある
- **3点**: 基本的なコスト意識はあるが、自動化や継続的最適化の仕組みが不足
- **2点**: コスト管理が受動的で、最適化の具体策がない
- **1点**: コスト管理への意識が全くない

---

## コスト見積もりのフレームワーク

### 良い見積もりに含まれる要素

1. **計算リソース**: サーバー / コンテナ / サーバーレスの実行コスト
2. **ストレージ**: データベース / オブジェクトストレージ / キャッシュ
3. **ネットワーク**: データ転送（egress）/ CDN / API Gateway
4. **サードパーティ**: SaaS / API呼び出し / ライセンス
5. **人件費**: 開発 / 運用 / 学習コスト
6. **成長予測**: トラフィック・データ量の増加に伴うコスト変動

---

## 良い回答例

> 「初期フェーズはVercel（無料枠）+ Supabase（無料枠）で月額$0で運用します。
> ユーザー1,000人を超えた段階で、Vercel Proプラン($20/月)に移行し、
> Supabaseも$25/月のProプランに切り替えます。
> 静的アセットはCloudflare CDN（無料）で配信し、オリジンへの負荷を80%削減します。
> API呼び出しはRedis（Upstash無料枠: 10,000リクエスト/日）でキャッシュし、
> DB負荷とコストを抑制します。月間1万ユーザー時の想定コストは$50-80/月です。
> 開発環境は夜間自動停止し、検証環境は必要時のみ起動する運用にします。」

## 悪い回答例

> 「AWSでEC2とRDSを使います。コストはあとで計算します。」
> → コスト見積もりの意識がなく、最適化の具体策もない
