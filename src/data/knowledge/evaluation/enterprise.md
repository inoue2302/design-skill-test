# 大企業条件での評価基準

## 概要

大企業特有の制約（ガバナンス、コンプライアンス、既存システム連携、大規模チーム）における
設計判断力を評価するための基準。技術的な正しさに加え、組織的・政治的な制約を考慮した
現実的な設計力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- 社内の技術標準・承認済みリストを確認し、準拠した技術選定を行っている
- 既存システムとの統合方式（API連携、ETL、イベント連携）を考慮した技術選定
- 長期サポート（LTS）のあるフレームワーク・ランタイムを優先している
- ベンダーロックインのリスクを認識し、抽象化層を設けて軽減
- 技術選定の意思決定プロセス（ADR: Architecture Decision Record）を文書化

### アンチパターン

- 社内標準を無視して最新技術を導入し、運用チームが対応できない
- ベンダー製品を無批判に採用し、カスタマイズ費用が膨大になる
- OSSの採用時にライセンスの確認を怠る（GPL汚染等）
- 既存システムとの連携を後回しにし、結合テストで大きな問題が発覚
- 技術選定の理由を文書化せず、将来の担当者が意図を理解できない

### スコアリング指針

- **5点**: 組織の制約・既存資産・長期運用を考慮し、ADRで判断を文書化
- **4点**: 社内標準を考慮した現実的な技術選定で、連携方式も明確
- **3点**: 技術的には妥当だが、組織的な制約への配慮が不足
- **2点**: 社内の既存システムやチーム体制を考慮していない
- **1点**: 組織の制約を全く考慮せず、理想論のみの技術選定

---

## 2. 設計力（architecture）

### 良い設計判断

- 段階的な移行計画（ストラングラーフィグパターン等）で既存システムからの移行を設計
- マイクロサービス/SOAの適用範囲をチーム構造（コンウェイの法則）に合わせて設計
- API Gatewayによる内部・外部APIの一元管理と認証の集約
- イベント駆動アーキテクチャで疎結合な部門間連携を実現
- 耐障害性パターン（サーキットブレーカー、バルクヘッド）を標準化

### アンチパターン

- ビッグバンリプレースを計画し、移行リスクが極大化
- 全システムを一度にマイクロサービス化しようとする
- 部門間のデータ共有をファイル転送（CSV, FTP）に依存
- 単一のモノリスに全部門の機能を詰め込む
- アーキテクチャのガバナンスがなく、各チームが独自に技術選定

### スコアリング指針

- **5点**: 既存システムとの共存・段階的移行・チーム構造を考慮した現実的なアーキテクチャ
- **4点**: 適切な粒度でのサービス分割と、移行計画の概要がある
- **3点**: アーキテクチャは妥当だが、既存システムとの移行戦略が不明確
- **2点**: 既存環境を無視したグリーンフィールド前提の設計
- **1点**: 大規模システムの設計経験・知識が不足

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- マスターデータ管理（MDM）の方針を定め、データの一元性を確保
- データオーナーシップの明確化とデータカタログの整備を計画
- 既存DBとの連携方式（CDC / API / バッチ）をデータ特性に応じて選択
- 個人情報保護法・GDPR等のコンプライアンス要件をデータ設計に反映
- 監査ログの設計で、誰が・いつ・何を変更したかを追跡可能

### アンチパターン

- 部門ごとにマスターデータが乱立し、データの不整合が常態化
- 個人情報の取り扱い基準を把握せず、違反リスクのあるデータフロー
- 既存DBのスキーマを無視して新システムを設計し、移行が困難
- データガバナンスの体制がなく、品質が維持されない
- 監査要件を後から追加し、大幅な改修が必要になる

### スコアリング指針

- **5点**: MDM・ガバナンス・コンプライアンス・監査を統合したデータ設計
- **4点**: データオーナーシップとコンプライアンスを考慮した設計
- **3点**: 基本的なデータ設計はできるが、ガバナンスの視点が不足
- **2点**: 企業のデータ管理要件を理解しておらず、リスクがある
- **1点**: データ設計の基本と企業要件の両方が欠如

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- SLA/SLOを既存の社内基準に合わせて定義し、達成手段を設計
- セキュリティ要件を社内のセキュリティポリシーに準拠させている
- 災害復旧（DR）計画をBCP（事業継続計画）と整合させている
- 性能テスト計画に本番相当のデータ量・トラフィックを含めている
- マルチテナント設計でのデータ分離とアクセス制御を厳格に実装

### アンチパターン

- 社内のセキュリティ審査プロセスを把握せず、リリース直前で指摘を受ける
- DR計画がなく、データセンター障害時の復旧手順が存在しない
- 性能テストを実施せず、本番リリース後にパフォーマンス問題が発覚
- アクセス権限の設計が粗く、部門をまたいだ不正アクセスが可能
- コンプライアンス要件（PCI DSS, SOC2等）を開発終盤で初めて確認

### スコアリング指針

- **5点**: 社内基準・コンプライアンス・DR・性能テストを包括的に設計し、審査を見据えている
- **4点**: セキュリティポリシー準拠とDR計画が含まれた設計
- **3点**: 基本的なセキュリティは考慮しているが、企業固有の要件への対応が不足
- **2点**: 社内基準の存在を認識しておらず、後から大幅な修正が必要
- **1点**: 非機能要件を全く考慮していない

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- 運用チーム（SRE / インフラチーム）との責任分界点を明確にしている
- ドキュメント（設計書、運用手順書、障害対応マニュアル）の作成計画がある
- 段階的なリリース計画で、影響範囲を限定しながらロールアウト
- TCO（Total Cost of Ownership）を初期費用と運用費用に分けて算出
- ベンダーとのSLA契約内容を把握し、サポート体制を確認

### アンチパターン

- 開発チームだけで設計を完結させ、運用チームへの引き継ぎがない
- ドキュメントを一切作成せず、属人的な知識に依存
- 全社一斉ロールアウトで、障害時の影響が全部門に及ぶ
- ライセンス費用の見積もりを忘れ、予算を大幅に超過
- ベンダー選定時にロックイン条項や契約の柔軟性を確認しない

### スコアリング指針

- **5点**: 運用移管・TCO算出・段階リリース・ドキュメントを包括した運用設計
- **4点**: 運用チームとの連携計画とコスト見積もりがある
- **3点**: 基本的な運用は考慮しているが、組織間連携やTCOの算出が不足
- **2点**: 運用設計が開発チーム内で閉じており、組織的な視点がない
- **1点**: 運用やコストへの配慮が全くない

---

## 良い回答例

> 「既存の基幹システムとはAPI Gateway経由で連携し、ストラングラーフィグパターンで
> 段階的に移行します。社内のJava標準に従いSpring Bootを採用し、ADRで選定理由を記録します。
> 個人情報はGDPR対応として暗号化とアクセスログを必須とし、監査証跡を残します。
> DRサイトを別リージョンに構築し、RPO 1時間・RTO 4時間を目標にします。
> 初期は営業部門のみに限定リリースし、3ヶ月かけて全社展開する計画です。
> TCOは初年度3,000万円、2年目以降は年間1,500万円を見込みます。」

## 悪い回答例

> 「Next.jsとSupabaseでモダンなアプリを作り、Vercelにデプロイします。」
> → 大企業の制約（既存連携、セキュリティ基準、組織体制）を全く考慮していない
