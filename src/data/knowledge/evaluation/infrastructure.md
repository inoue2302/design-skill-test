# インフラ設計の評価基準

## 概要

クラウドインフラストラクチャの設計力を評価するための基準。
可用性・拡張性・セキュリティ・コスト効率を総合的に判断する能力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- クラウドプロバイダー（AWS / GCP / Azure）の選定理由を要件から説明できる
- コンテナオーケストレーション（ECS / EKS / Cloud Run）をワークロードに応じて選択
- IaC ツール（Terraform / Pulumi / CDK）の採用理由を運用体制から説明できる
- サーバーレス vs コンテナ vs VM の判断基準が明確

### アンチパターン

- 「AWSしか知らないから」という理由だけでプロバイダーを選定
- Kubernetesが不要な規模で無理にEKSを導入する
- IaCを使わず、すべてコンソールから手動設定
- マネージドサービスの存在を知らず、自前で構築しようとする

### スコアリング指針

- **5点**: ワークロード特性・チーム体制・コストを総合判断し、最適なインフラ構成を選定
- **4点**: 適切なサービスを選定し、主要なトレードオフを説明できる
- **3点**: 一般的なクラウド構成を選定できるが、代替案の検討が不足
- **2点**: 特定のサービスに偏った選定で、要件との適合性が低い
- **1点**: クラウドサービスの基本的な理解が不足している

---

## 2. 設計力（architecture）

### 良い設計判断

- マルチAZ構成で単一障害点を排除している
- ネットワーク設計でVPC・サブネット・セキュリティグループを適切に分離
- CDN / ロードバランサー / オートスケーリングの組み合わせが合理的
- CI/CDパイプラインがインフラのプロビジョニングも含めて自動化されている
- 環境分離（dev / staging / prod）がネットワークレベルで実現されている

### アンチパターン

- 単一AZ構成で可用性を犠牲にしている
- セキュリティグループのルールが過度に緩い（0.0.0.0/0 を多用）
- パブリックサブネットにデータベースを配置している
- 手動でのインフラ変更が常態化し、設定の再現性がない
- ステージング環境が本番と大きく乖離している

### スコアリング指針

- **5点**: 耐障害性・セキュリティ・拡張性を多層的に設計し、構成図で説明できる
- **4点**: 基本的なHA構成とネットワーク分離を実現
- **3点**: 標準的な構成だが、障害シナリオへの備えが不十分
- **2点**: 構成に明らかな脆弱性があり、可用性に不安
- **1点**: インフラ設計の基本概念が欠如

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- データベースの冗長構成（リードレプリカ、マルチマスター）を要件に応じて設計
- バックアップ戦略（RPO / RTO）を事業要件から逆算して定義
- ストレージ階層（ホット / ウォーム / コールド）を使い分けてコスト最適化
- データの暗号化（at rest / in transit）を適切なレイヤーで実装
- ログ・メトリクス・トレースのデータパイプラインを設計している

### アンチパターン

- バックアップの取得頻度やリストア手順が未定義
- すべてのデータを同一のストレージクラスに保存
- データベースのスナップショットだけに依存し、論理バックアップがない
- ログの保持期間やローテーションが未設計
- 機密データの暗号化方式が不明確

### スコアリング指針

- **5点**: RPO/RTOに基づくバックアップ設計、暗号化、ストレージ最適化が統合されている
- **4点**: バックアップと冗長構成が適切で、データ保護の基本を満たす
- **3点**: バックアップは取得しているが、リストア手順やRPO/RTOの定義が不足
- **2点**: データ保護の設計が断片的で、災害復旧の計画がない
- **1点**: バックアップやデータ保護の概念がない

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- SLA/SLOを定義し、それを達成するためのインフラ構成を設計
- DDoS対策（WAF / Shield / レート制限）を多層防御で実装
- ゼロトラストネットワークの原則に基づいたアクセス制御
- 障害検知から復旧までの自動化（セルフヒーリング）を設計
- パフォーマンステストの環境とシナリオを事前に計画

### アンチパターン

- SLAを定義せず、「落ちたら直す」という対応方針
- WAFを導入せず、アプリケーション層のみでセキュリティを担保しようとする
- IAMロールの権限が過剰（AdministratorAccess を多用）
- 障害時のエスカレーションフローが未定義
- キャパシティプランニングをせず、リソースの過不足が頻発

### スコアリング指針

- **5点**: SLO定義、多層防御、自動復旧、キャパシティプランニングを包括的に設計
- **4点**: 可用性目標とセキュリティ対策が明確で、監視体制も整備
- **3点**: 基本的なセキュリティは考慮しているが、可用性設計が不十分
- **2点**: 非機能要件の考慮が表面的で、具体的な数値目標がない
- **1点**: 非機能要件を全く考慮していない

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- コスト見積もりをリソース単位で算出し、月額の概算を提示できる
- Reserved Instances / Savings Plans / スポットインスタンスを適切に活用
- CloudWatch / Datadog 等の監視設計で、アラート閾値を段階的に設定
- インフラのコード化（IaC）でドリフト検知と自動修復を実現
- コストアラートを設定し、予算超過を事前に検知する仕組み

### アンチパターン

- コスト見積もりをせず、月末の請求書で初めて把握する
- すべてのインスタンスをオンデマンドで稼働させている
- 監視が死活監視のみで、パフォーマンスの劣化を検知できない
- 不要なリソース（停止し忘れたインスタンス等）が放置されている
- タグ付けが不十分で、コストの按分ができない

### スコアリング指針

- **5点**: コスト最適化・監視・自動化・IaCが統合された運用設計
- **4点**: コスト見積もりと基本的な監視・アラート設計がある
- **3点**: 運用の基本は考慮しているが、コスト最適化の具体策が不足
- **2点**: 運用設計が属人的で、自動化が進んでいない
- **1点**: 運用やコストへの配慮が全くない

---

## 良い回答例

> 「AWSのマルチAZ構成で、ECS Fargateを採用します。コンテナ化することで環境の再現性を担保し、
> Fargateでサーバー管理を不要にします。RDSはMulti-AZで冗長化し、RPO 1時間・RTO 4時間を
> 目標にバックアップを設計します。Terraformで全インフラをコード化し、GitHub Actions経由で
> 適用します。月額コストは$200-300を想定し、Savings Plansで30%削減を見込みます。」

## 悪い回答例

> 「EC2にデプロイして、RDSを使います。」
> → 構成の詳細がなく、可用性・コスト・運用の考慮が欠如
