# モバイル設計の評価基準

## 概要

モバイルアプリケーション設計における技術的判断力を評価するための基準。
ネイティブ / クロスプラットフォーム / PWA の選定から、モバイル特有の制約を考慮した設計力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- ネイティブ（Swift/Kotlin）vs クロスプラットフォーム（Flutter/React Native）の選定理由が明確
- PWA の適用可能性を検討し、ネイティブアプリが本当に必要かを判断している
- BaaS（Firebase / Supabase）の活用可否をチーム規模・要件から判断
- プッシュ通知・位置情報・カメラ等のネイティブ機能の要件を整理して技術選定に反映
- 状態管理ライブラリの選定がアプリの複雑度に見合っている

### アンチパターン

- 「iOSとAndroid両方必要だから」だけでクロスプラットフォームを選定する
- ネイティブ機能を多用するのにWebViewベースのフレームワークを選択
- アプリ配布の審査プロセス（App Store Review / Google Play Policy）を考慮しない
- バックエンドの技術選定がモバイル特有の通信パターンを考慮していない
- 画像・動画処理の要件があるのにパフォーマンス特性を検証しない

### スコアリング指針

- **5点**: 要件・制約・チーム構成・ストア審査を総合判断し、最適なアプローチを選定
- **4点**: 適切な技術を選定し、モバイル特有の考慮事項を把握している
- **3点**: 一般的に妥当な選定だが、モバイル固有の制約への考慮が不足
- **2点**: 選定理由が曖昧で、モバイル特有の要件を見落としている
- **1点**: モバイル技術の基本的な選択肢を理解していない

---

## 2. 設計力（architecture）

### 良い設計判断

- アーキテクチャパターン（MVVM / Clean Architecture / BLoC）を適切に選択
- オフラインファースト設計で、ネットワーク切断時もアプリが動作する
- 画面遷移のフロー設計がユーザー体験を考慮し、Deep Link にも対応
- モジュール分割でビルド時間を短縮し、チーム開発を効率化
- バックグラウンド処理（同期・通知・位置情報）のライフサイクル管理を設計

### アンチパターン

- ViewControllerやActivityにビジネスロジックが集中（Massive VC問題）
- ネットワーク接続を前提とし、オフライン時にクラッシュする
- 画面遷移がスパゲッティ状態で、バックスタックの管理が破綻
- 全機能を単一モジュールに詰め込み、ビルド時間が長大
- メモリリークの原因となる循環参照を考慮していない

### スコアリング指針

- **5点**: クリーンアーキテクチャ・オフライン対応・モジュール設計が統合された堅牢な構造
- **4点**: 適切なアーキテクチャパターンを採用し、主要な設計原則に従っている
- **3点**: 基本的な画面設計はできるが、オフライン対応やモジュール化が不十分
- **2点**: 設計パターンが不明確で、責務の分離ができていない
- **1点**: アーキテクチャ設計の概念が欠如

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- ローカルDB（SQLite / Realm / Hive）の選定がデータ特性に基づいている
- オフラインデータの同期戦略（楽観的ロック / CRDT / Last Write Wins）を明確にしている
- API通信のキャッシュ戦略をデータの鮮度要件に応じて設計
- 画像・動画のキャッシュ・圧縮戦略でストレージとメモリを最適化
- Secure Storage（Keychain / Keystore）による機密データの保護

### アンチパターン

- すべてのデータをUserDefaults / SharedPreferencesに保存する
- ローカルキャッシュの無効化戦略がなく、古いデータが残り続ける
- APIレスポンスをそのままローカルDBに保存し、スキーマ変更で壊れる
- 認証トークンをプレーンテキストで保存する
- 大量のデータをメモリに展開し、OOM（Out of Memory）を引き起こす

### スコアリング指針

- **5点**: オフライン同期・キャッシュ戦略・セキュアストレージを包括的に設計
- **4点**: ローカルDBとキャッシュの設計が適切で、セキュリティも考慮
- **3点**: 基本的なデータ保存はできるが、同期やセキュリティの設計が不足
- **2点**: データ管理の方針が不明確で、整合性に問題がある
- **1点**: モバイルのデータ設計に関する知識が不足

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- アプリサイズの最適化（コード分割、アセット圧縮、On-Demand Resources）を設計
- バッテリー消費を考慮した設計（バックグラウンド処理の最適化、位置情報の精度調整）
- 多様なデバイス（画面サイズ、OS バージョン、メモリ量）への対応を設計
- アクセシビリティ（VoiceOver / TalkBack 対応）を要件に含めている
- アプリ起動時間の目標値を設定し、Cold Start / Warm Start を最適化

### アンチパターン

- 最新OSバージョンのみをサポートし、古い端末のユーザーを切り捨てる
- 高精度GPS を常時取得し、バッテリーを急速に消耗させる
- 画像を圧縮せず転送・表示し、通信量とメモリを浪費
- アクセシビリティを一切考慮せず、スクリーンリーダーで操作できない
- ANR（Application Not Responding）の原因となるメインスレッドのブロック

### スコアリング指針

- **5点**: パフォーマンス・バッテリー・アクセシビリティ・デバイス対応を網羅的に設計
- **4点**: 主要な非機能要件を特定し、対策を設計に組み込んでいる
- **3点**: パフォーマンスは意識しているが、バッテリーやアクセシビリティの考慮が不足
- **2点**: 非機能要件への言及が少なく、具体的な対策がない
- **1点**: 非機能要件を全く考慮していない

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- CI/CD（Fastlane / GitHub Actions / Bitrise）でビルド・テスト・配布を自動化
- クラッシュ分析（Crashlytics / Sentry）を導入し、品質を継続的に監視
- A/Bテスト・Feature Flag でリリースリスクを低減
- ストア審査のリードタイムを考慮したリリース計画
- CodePush / OTA アップデートの適用可否を判断している

### アンチパターン

- 手動でビルド・署名・アップロードしており、属人化している
- クラッシュレポートがなく、ユーザーからの報告でしか障害を検知できない
- 強制アップデートの設計がなく、古いバージョンのサポートが際限なく続く
- ストア審査でリジェクトされるリスクを事前に検討していない
- バックエンドのAPI変更時にアプリのバージョン互換性を考慮しない

### スコアリング指針

- **5点**: CI/CD・クラッシュ監視・段階的リリース・バージョン管理を統合した運用設計
- **4点**: 自動化された配信パイプラインと基本的な監視体制
- **3点**: 基本的なビルド自動化はあるが、監視やリリース戦略が不十分
- **2点**: 運用が手動中心で、品質管理の仕組みがない
- **1点**: 運用やコストへの配慮が全くない

---

## 良い回答例

> 「React Nativeを採用します。理由は、iOS/Android両対応が必要で、チームにTypeScript経験者が
> 多いからです。オフラインファースト設計でWatermelonDBをローカルDBに使い、楽観的更新で
> 同期します。認証トークンはSecure Storageに保管し、画像はFastImageでキャッシュ管理します。
> Fastlane + GitHub Actionsで配信を自動化し、CrashlyticsでクラッシュをリアルタイムPIM監視します。
> 初回起動は2秒以内を目標にし、バンドルサイズを15MB以下に抑えます。」

## 悪い回答例

> 「Flutterで作ります。Firebaseでバックエンドも賄います。」
> → オフライン対応・セキュリティ・パフォーマンス目標の考慮が欠如
