# Webアプリ設計の評価基準

## 概要

Webアプリケーション設計における技術的判断力を評価するための基準。
フロントエンド・バックエンド・インフラの横断的な設計力を測定する。

---

## 1. 技術選定力（stackSelection）

### 良い設計判断

- 要件に合ったフレームワークを選定し、その理由を明確に説明できる
- SSR / SSG / CSR / ISR の使い分けを正しく理解し、ユースケースに応じて選択している
- 認証・認可ライブラリをセキュリティ要件に基づいて選定している
- 状態管理の手法（Server State vs Client State）を適切に区別している

### アンチパターン

- 「流行っているから」「前に使ったから」だけで技術を選定する
- SPAが不要な場面で重厚なクライアントフレームワークを導入する
- ORM選定時にクエリパフォーマンスやマイグレーション戦略を考慮しない
- フロントエンドの状態管理にグローバルストアを安易に多用する

### スコアリング指針

- **5点**: 要件・制約・チーム構成を総合的に判断し、トレードオフを明示して技術を選定
- **4点**: 適切な技術を選定し、主要な理由を説明できる
- **3点**: 一般的に妥当な選定だが、トレードオフの考慮が不十分
- **2点**: 選定理由が曖昧、または要件とのミスマッチがある
- **1点**: 技術選定の根拠がなく、要件を満たせない可能性が高い

---

## 2. 設計力（architecture）

### 良い設計判断

- レイヤー分離（Presentation / Application / Domain / Infrastructure）が明確
- コンポーネント設計において責務の分離と再利用性を両立している
- API設計でRESTful原則またはGraphQLの特性を適切に活用している
- エラーハンドリング戦略が一貫しており、ユーザー体験を損なわない

### アンチパターン

- ページコンポーネントにビジネスロジックを直接記述する
- APIエンドポイントの粒度が粗すぎる、または細かすぎる
- 認証・認可のチェックが散在し、漏れが発生しやすい構造
- フロントとバックエンドの責務境界が曖昧

### スコアリング指針

- **5点**: 変更容易性・テスタビリティ・スケーラビリティを考慮した堅牢なアーキテクチャ
- **4点**: 主要な設計原則に従い、拡張性のある構造
- **3点**: 基本的な分離はできているが、一部に密結合がある
- **2点**: 設計の意図が不明確で、変更時の影響範囲が広い
- **1点**: モノリシックで責務の分離がほぼない

---

## 3. データ設計力（dataDesign）

### 良い設計判断

- ER図レベルでエンティティの関係性を説明できる
- 正規化と非正規化の判断基準を読み書き比率やアクセスパターンから導いている
- キャッシュ戦略（CDN / アプリケーションキャッシュ / DBキャッシュ）を階層的に設計している
- セッション管理・トークン管理の方式を安全性と利便性の観点で設計している

### アンチパターン

- すべてのデータを単一テーブルに格納する
- N+1問題を考慮せずにORMのリレーションを定義する
- キャッシュの無効化戦略を考慮せずにキャッシュを導入する
- ユーザー入力のバリデーションをフロントエンドのみで行う

### スコアリング指針

- **5点**: アクセスパターンに基づいた最適なデータモデルを設計し、整合性とパフォーマンスを両立
- **4点**: 正規化・インデックス設計が適切で、主要なクエリパターンを考慮
- **3点**: 基本的なテーブル設計はできるが、パフォーマンス面の考慮が不足
- **2点**: データの関係性の定義が不十分で、整合性に問題がある
- **1点**: データ設計の基本概念が理解できていない

---

## 4. 非機能要件の考慮（nonFunctional）

### 良い設計判断

- Core Web Vitals（LCP, FID, CLS）を意識したフロントエンド設計
- レスポンスタイムの目標値を設定し、それを達成するための設計を行っている
- セキュリティ（XSS, CSRF, SQLi, SSRF）への対策を設計段階で組み込んでいる
- アクセシビリティ（WCAG準拠レベル）を要件に応じて考慮している

### アンチパターン

- パフォーマンスを後回しにし、機能開発のみに集中する
- HTTPS化やCSPヘッダーなどの基本的なセキュリティ対策を忘れる
- モバイルファーストを考慮せず、デスクトップ前提で設計する
- 負荷テストの計画がなく、本番トラフィックに対する備えがない

### スコアリング指針

- **5点**: パフォーマンス・セキュリティ・可用性・ユーザビリティを包括的に設計
- **4点**: 主要な非機能要件を特定し、対策を設計に組み込んでいる
- **3点**: 一部の非機能要件は考慮しているが、網羅性に欠ける
- **2点**: 非機能要件への言及が少なく、具体的な対策がない
- **1点**: 非機能要件を全く考慮していない

---

## 5. 運用・コスト意識（operationCost）

### 良い設計判断

- CI/CDパイプラインの設計を含め、デプロイ戦略を明確にしている
- ログ・モニタリング・アラートの設計を運用フェーズから逆算している
- Vercel / Cloudflare 等のエッジプラットフォームを活用してコストを最適化
- Feature Flagを活用した段階的リリース戦略を提案できる

### アンチパターン

- ログ出力の設計がなく、障害時の原因調査が困難
- 環境ごとの設定管理（dev / staging / prod）が考慮されていない
- デプロイ手順がマニュアルで属人化している
- 無料枠の制約を把握せず、予期せぬ課金が発生する設計

### スコアリング指針

- **5点**: 運用コスト・チーム体制・将来のスケールを見据えた現実的な運用設計
- **4点**: CI/CD・監視・ログを設計に含め、運用負荷を考慮
- **3点**: 基本的なデプロイ方法は考えているが、監視やコスト管理が不十分
- **2点**: 運用フェーズの設計がほぼなく、開発完了=リリースという発想
- **1点**: 運用についての考慮が一切ない

---

## 良い回答例

> 「Next.js App Routerを採用します。理由は、SEOが重要なページにはSSRを、ダッシュボードにはCSRを使い分けられるからです。
> 認証にはNextAuth.jsを使い、JWTとセッションの両方をサポートします。
> データベースはPlanetScaleを選定し、ブランチ機能でスキーマ変更の安全性を確保します。
> Core Web Vitalsの目標値を設定し、Lighthouseスコア90以上を維持する設計にします。」

## 悪い回答例

> 「ReactとNode.jsで作ります。データベースはMySQLを使います。」
> → 選定理由がなく、非機能要件やアーキテクチャへの言及がない
